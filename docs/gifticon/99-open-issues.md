# 99. 미정 / 결정 필요 항목

이 문서는 결정될 때마다 해당 섹션이 정식 문서로 옮겨가며 비워진다.

---

## A. 수신번호 등록 / 인증 정책

기프티콘은 발급되면 **환불 사실상 불가**라 오발송 리스크가 큼.

### 결정 필요 사항

1. 매번 입력 vs 한 번 등록 후 사용
2. 등록 시 SMS OTP 인증 받을지
3. 등록된 번호로만 발송 가능 vs 다른 번호도 가능 (선물 기능)

### 후보 조합

| 조합 | 안전성 | UX | 선물 |
|---|---|---|---|
| 인증 + 본인만 | 최고 | 최초 1회 인증 필요 | 불가 |
| 인증 + 자유 입력 | 보통 | 매번 번호 입력 | 가능 |
| 입력만 + 본인 강제(가입 번호) | 보통 | 빠름 | 불가 |
| 입력만 + 자유 입력 | 낮음 | 빠름 | 가능 |

### 메모

- 캐시모어가 이미 본인 인증 인프라를 쓰는지 확인 필요
- "한 번 등록"은 합의됨 (사용자 확인). 인증 여부 / 본인 강제 여부는 별도 논의

## B. 결제 흐름의 atomicity

### 시나리오

포인트 차감 → `couponCreate.sc` 호출 사이/응답 직전에 문제 발생 시:

1. **타임아웃** — 발송 됐는지 안 됐는지 모름
2. **서버 크래시** — 트랜잭션 커밋 후 외부 호출 직전에 죽으면 `pending` 영구 잔류
3. **클라이언트 중복 클릭** — 같은 주문 두 번 들어옴

### 후보 처리 방안

- 타임아웃 시 즉시 `couponState.sc`로 상태 조회 후 분기
  - 스마트콘에 주문 있음 → `sent` 확정
  - 주문 없음 → 환불 후 `send_failed`
  - 조회도 실패 → `pending` 유지, 배치가 재확인
- `pending` 상태 잔류 건은 N분마다 배치가 `couponState.sc`로 폴링
- 클라이언트 idempotency_key를 받아 중복 차단 (TR_ID 우리가 생성하므로 별도 필요)

## F. 상태 폴링 정책

스마트콘은 webhook 없음 — 우리가 `couponState.sc`로 polling 해야 사용/취소 상태 알 수 있음.

### 옵션

- 배치 주기적 폴링 (`pending`/`sent` 상태인 주문 대상)
- 사용자가 내역 화면 진입할 때 lazy 호출
- 둘 다 (베이스라인 배치 + 사용자 요청 시 즉시 갱신)

### 폴링 종료 조건

- `EXCHANGE_STATUS = 1` (교환 완료)
- 유효기간 만료
- `CLAIM_TYPE = CC` (주문취소)

## G. 포인트 정책

- 1P = 1원 환산 가정해도 되는지 (캐시모어 내 다른 기능 기준)
- 최소 차감 단위 / 보유 한도 검증
- 차감 트랜잭션은 기존 `point_actions` 패턴 그대로 (cash_exchanges 패턴 참고)

## H. 스마트콘 dev 환경 / 응답 스펙 차이 (2026-05-04 검증)

ECS 운영 컨테이너에서 `GetEventGoods.sc` 직접 호출하여 검증한 결과.

### 환경별 호출 결과

| 환경 | 결과 |
|---|---|
| **prd** (`b2b-api-prd...`) | ✅ 200 OK, JSON 응답, EVENT_ID 64385로 상품 2개 수신 |
| **dev** (`b2b-api-dev...`) | ❌ connection timeout |

→ dev 미동작 원인 후보: ① EVENT_ID 64385가 dev에 미등록, ② NAT IP가 dev 방화벽에 미등록.
→ 스마트콘 담당자에게 dev 환경 확인 요청 필요. 그 전까지는 개발 시에도 prd 호출하거나, 응답 fixture로 대체.

### EVENT 64385 라인업

현재 스마트콘 측에서 EVENT 64385에 **2개 상품만 노출되도록 세팅된 상태** (사용자 확인). 응답 자체는 정상. 운영 확장 시 사업·스마트콘 협의로 라인업 추가 필요. 개발 측에서는 받은 응답 기준으로 진행.

### 응답 스펙 차이 (V1.9 문서 vs 실제)

**문서엔 없는데 실제 응답에 있는 필드** (모두 `smartcon_goods` 컬럼화함):

- `IMG_URL_HTTPS` — HTTPS 이미지 (모바일 노출용, **이게 진짜 노출용**)
- `EXTRA_CHARGE` — 추가 요금 (예: ICE 변경 1000원)
- `GOODS_SALE_TYPE` — 발급 방식 (`"BARCODE"` 등)
- `GOODS_USE_TYPE` — 사용 타입 (`"EXCHANGE"` 등)
- `SC_LIMIT_DATE` — 유효기간 일수 (예: 30) — **의미 확인 필요**
- `B2C_ITEM_NO` — 현재 모두 null

**문서엔 있는데 응답에 없는 필드** (옵션이라 제외, `raw_data`에서 향후 추출 가능):

- `LIMIT_CNT`
- `SAMSUNGPAY_RATE`

### 결정 필요

- `SC_LIMIT_DATE`의 정확한 의미 (발송일로부터 N일? 발급일로부터? 다른 기준?)
- `EXTRA_CHARGE > 0` 상품을 노출할 것인지 / 노출 시 UX 처리
- dev 환경 사용 가능하게 만들 것인지 / fixture로 갈 것인지
