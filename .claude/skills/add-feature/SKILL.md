# 기존 모듈에 기능 추가

이미 있는 모듈에 새 엔드포인트, 메서드, 또는 기능을 추가할 때의 워크플로우.

## Step 1: 인터뷰

기능을 추가하기 전에 반드시 사용자에게 다음을 확인한다. `$ARGUMENTS`에 이미 답이 포함되어 있으면 해당 항목은 건너뛴다.

1. **대상 모듈**: 어떤 모듈에 추가하는지 (예: `point`, `attendance`, `invitation`)
2. **추가할 기능**: 무엇을 추가하는지 (한 문장)
3. **새 엔드포인트 필요 여부**: HTTP API가 추가되는지, 아니면 Service 메서드만 추가하는지
4. **새 DB 쿼리 필요 여부**: Repository 인터페이스에 메서드를 추가해야 하는지
5. **새 테이블/컬럼 필요 여부**: DB 스키마 변경이 필요한지

인터뷰가 끝나면 "이렇게 추가하겠습니다"라고 요약을 보여주고 확인을 받는다.

## Step 2: 현재 코드 파악

대상 모듈의 기존 코드를 먼저 읽는다:
- Interface (어떤 메서드가 이미 있는지)
- Service (비즈니스 로직 흐름)
- Controller (기존 엔드포인트 패턴)

## Step 3: 코드 수정

기존 모듈의 코드 스타일과 패턴을 따라서 수정한다.

## Step 4: 체크리스트

수정 후 아래 체크리스트를 확인한다. 빠뜨린 항목이 있으면 추가한다.

### Repository 메서드를 추가한 경우

- [ ] Interface에 새 메서드 시그니처 추가 (+ 필요한 도메인 타입)
- [ ] Supabase Repository에 구현 추가
- [ ] Stub Repository에 구현 추가 (Map 조회/필터링 로직 + 테스트 setter)
- [ ] 단위 테스트에 새 메서드 관련 테스트 추가
- [ ] E2E 테스트에 새 기능 관련 테스트 추가

### 엔드포인트를 추가한 경우

- [ ] Controller에 새 엔드포인트 추가 (Swagger 데코레이터 포함)
- [ ] DTO 파일 추가/수정 (Request + Response)
- [ ] E2E 테스트에 새 엔드포인트 테스트 추가
- [ ] 인증 필요 시 Guard + 인증 실패 테스트 추가

### Service 메서드만 추가한 경우

- [ ] 단위 테스트에 새 메서드 관련 테스트 추가

### 테스트 헬퍼

- [ ] E2E 테스트에서 새로 필요한 테스트 데이터가 있으면 헬퍼 함수 추가/수정

## Step 5: 테스트 실행

```bash
bun test src/{module}              # 단위 테스트
bun test test/{module}.e2e.spec.ts  # E2E 테스트
```

기존 테스트가 깨지지 않았는지 반드시 확인한다.
